## Información del Departamento (Notion)

### Objetivo de la Práctica
Identificar atributos cuantitativos para las estimaciones de los requisitos del proyecto para determinar el esfuerzo, el coste y el calendario del proyecto

### Resumen de la Subpráctica
No se puede estimar bien el coste de un proyecto si no se tiene en cuenta el trabajo: Número de requisitos, Experiencia del equipo, Puntos por función. Considerando: Cada atributo debe tener un nivel relativo de dificultad, las estimaciones deben ser consistentes con los requisitos del proyecto, se deben usar modelos validados o datos históricos, se debe definir primero la aproximación técnica.

### Artefactos que Cumplen con la Práctica
- RTM - Requirement Traceability Matrix (define todas las historias de usuario y desglosa su complejidad)
- SRS - Software Requirement Specification (establece el número de requisitos funcionales)
- SAD - Software Architecture Document (establece la arquitectura y la dificultad de implementación)
- WBS - Work Breakdown Structure (establece toda la descomposición de los entregables para completar un proyecto)
- Backlog (seguimiento de las historias de usuario)

### Proyectos de Referencia
UNAM (WBS, SRS, User Story Mapping, RTM, Cronograma de Trabajo, Plan de trabajo, PVG), GeekDesign (RTM), Poder Judicial (WBS, SRS, User Story Mapping, RTM, Cronograma de Trabajo, PVG, Plan de Trabajo), Excalibur (RTM, WBS, SRS, User Story Mapping, Cronograma de Trabajo, Plan de Trabajo, PVG), Departamento
